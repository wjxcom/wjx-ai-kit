# 用户画像数据方案与评估指标设计（草案）

关联 Issue：CMP-7《用户画像数据方案与评估指标设计》
适用场景：问卷星平台智能问卷推荐引擎的用户画像与评估体系设计

---

## 1. 用户画像构建

### 1.1 用户特征维度定义

从「账号信息 + 使用行为 + 问卷内容偏好」三大块定义用户画像维度：

1. 账号基础属性
   - 行业（industry）：如互联网、教育、制造、政府、医疗等
   - 企业规模（company_size）：按员工数或收入分层（1-10人、11-50人、51-200人、200+ 等）
   - 地域（region）：省份 / 城市 / 国家
   - 角色类型（user_role）：市场、HR、教学教务、产品运营、研究人员等

2. 使用行为属性
   - 使用频率（usage_frequency）：
     - 指标：最近 30/90 天登录次数、活跃天数
     - 标签：高频（>= 3 天/周）、中频（1-2 天/周）、低频（< 1 天/周）
   - 创建问卷行为
     - 创建问卷数（created_survey_cnt）
     - 有效问卷数（publish_and_collect_cnt）
     - 模板使用比例（template_usage_ratio = 使用模板创建数 / 总创建数）
   - 回收与管理行为
     - 平均回收样本量（avg_response_cnt）
     - 数据导出频次（export_cnt）
     - 使用分析功能的深度（是否使用交叉分析、数据透视、图表导出等）

3. 问卷内容与偏好
   - 偏好问卷类型（preferred_survey_type）：
     - 市场调研、员工满意度、课程反馈、客户满意度、NPS 调研、问卷报名等
     - 从问卷标签 / 分类字段 / 模板类型中抽取
   - 题型偏好
     - 单选、多选、矩阵量表、NPS、打分题等的占比
   - 逻辑复杂度
     - 是否使用逻辑跳转、引用题、题目随机、选项随机等
   - 收集渠道偏好
     - 链接、二维码、微信公众号、小程序、短信、邮件等

> 最终用户画像以「特征向量」形式存储，如：
> - 分类特征：行业、角色、偏好问卷类型等 → one-hot / embedding
> - 数值特征：最近 30 天创建问卷数、平均样本量等 → 标准化或分箱

### 1.2 特征工程方案

1. 原始数据来源（假设）
   - `user` 表：`user_id`, `industry`, `company_size`, `region`, `role`, `register_time`, `plan_type` 等
   - `survey` 表：`survey_id`, `user_id`, `template_id`, `category`, `created_at`, `published_at`, `status` 等
   - `survey_question` 表：`question_id`, `survey_id`, `question_type`, `has_logic`, `position` 等
   - `survey_response` 表：`response_id`, `survey_id`, `user_id`, `submit_time`, `channel` 等
   - `event_log` 表：埋点行为，如登录、点击分析、导出、分享等

2. 特征提取示例
   - 使用频率：
     - `login_days_30d` = 最近 30 天有登录行为的天数
     - `active_days_90d` = 最近 90 天有任何事件的天数
   - 创作能力与习惯：
     - `created_surveys_90d` = 最近 90 天创建问卷数量
     - `published_ratio_90d` = 最近 90 天发布问卷数 / 创建问卷数
     - `template_usage_ratio_90d` = 最近 90 天使用模板创建问卷数 / 创建问卷数
   - 问卷复杂度：
     - `avg_questions_per_survey` = 最近 N 个问卷的平均题目数量
     - `logic_used_ratio` = 使用过逻辑设置的问卷数 / 总问卷数
   - 偏好类型与场景：
     - `preferred_category_top1` / `top2`：基于最近 N 个问卷的类别分布
     - `channel_distribution`：各收集渠道占比（可用 embedding 或聚合标签表示）

3. 数据清洗与标准化
   - 缺失值处理：
     - 行业/规模缺失时可标记为 "unknown"；也可基于同企业或同域名推断
   - 异常值：
     - 过滤极端使用情况（如爬虫账号），可通过行为规则+阈值识别
   - 特征缩放：
     - 对长尾分布的特征（问卷数、样本数等）使用 log 变换或分箱（0/1-5/6-20/20+）

### 1.3 用户分群策略

1. 规则分群（便于解释与运营）
   - 基于使用频率 + 创建问卷数 + 样本量：
     - 高价值重度用户：高频登录 + 高问卷数 + 高样本量
     - 增长潜力用户：中频 + 部分场景高使用，但整体问卷数不高
     - 休眠/流失风险用户：低频 + 最近 90 天无核心行为
   - 基于场景与类型：
     - HR 场景用户：员工满意度、360 评估、招聘相关问卷占比高
     - 教学教务用户：课程评估、教学反馈占比高
     - 市场运营用户：市场调研、活动报名、NPS 调研占比高

2. 无监督分群（用于推荐系统特征）
   - 通过 k-means / GMM / 层次聚类，对用户画像特征进行聚类
   - 聚类特征：活跃度、问卷复杂度、偏好类型、多渠道使用情况等
   - 聚类结果作为 `user_cluster_id` 特征，参与推荐模型训练

3. 标签体系
   - 为运营 / 策略侧输出标签字典：如
     - `tag_active_heavy_creator`：高活跃高创作者
     - `tag_hr_satisfaction_scenario`：HR 满意度场景高使用
     - `tag_risk_churn`：高流失风险

## 2. 评估指标体系

### 2.1 离线指标（模型效果）

针对推荐模型，基于历史日志构造「离线评估样本」：
- 用户画像 + 当时候选模板集合
- 真实被点击/使用的模板作为正样本

常用离线指标：

1. Precision@K
   - 定义：在推荐列表前 K 个结果中，真实被用户选择/使用的模板占比
   - 含义：推荐前排结果的准确程度

2. Recall@K
   - 定义：真实被使用的模板中，有多少比例出现在推荐列表前 K 个结果中
   - 含义：模型对「应推荐内容」的覆盖能力

3. NDCG@K（Normalized Discounted Cumulative Gain）
   - 定义：考虑位置权重的排序质量指标
   - 适用：当有多级反馈信号（点击、使用、完成）时，可设置不同增益分数

4. MAP（Mean Average Precision）
   - 面向多样候选集合时的整体检索效果评价

> 离线评估周期：
> - 新模型/特征方案上线前，在最近 1-3 个月历史数据上离线评估
> - 周期性（如每周）对模型进行回归测试，监控指标波动

### 2.2 在线指标（运营与业务）

上线后的在线评估关注真实用户行为：

1. CTR（Click-Through Rate）
   - 定义：推荐入口曝光后产生点击行为的比例
   - 维度：整体 CTR / 不同入口 CTR / 不同用户分群 CTR

2. 转化率（Conversion Rate）
   - 定义：从点击推荐模板到创建并发布问卷的转化比例
   - 可细分：点击 → 创建、创建 → 发布、发布 → 收集达到阈值

3. 模板使用完成率
   - 定义：从「选择模板」到「完成编辑并发布」的比例
   - 关注点：推荐是否真的帮助用户节省时间，而不是增加试错成本

4. 用户满意度指标
   - 主观反馈：
     - 在推荐区域加「本次推荐是否有帮助」的反馈入口，收集 👍/👎 或 1-5 分评价
   - 客观 Proxy：
     - 推荐模板被多次重复使用
     - 用户在推荐入口停留时间下降（说明更快找到合适模板）

### 2.3 指标分层与看板

- 按用户分群、行业、场景维度拆分 CTR / 转化率
- 对比不同推荐策略（随机 / 热门 / AI）的指标差异
- 构建 DataStudio / BI 看板，支持：
  - 总览（全站）
  - 分行业 / 场景 / 用户分群的 Drill-down

## 3. A/B 测试方案

### 3.1 实验分组策略

- 实验单位：`user_id`
- 分流方式：对用户 ID 做 hash，确保同一个用户在实验期间进入稳定分组
- 示例：
  - 10% 用户进入随机推荐组（Baseline-1）
  - 10% 用户进入热门推荐组（Baseline-2）
  - 20% 用户进入 AI 推荐组（Treatment）
  - 其余 60% 保持现有逻辑（观察对整体业务的影响）

### 3.2 对照组设计

- 随机推荐：在同一分类下随机选择模板，作为最简单基准
- 热门推荐：基于全站/同场景模板的历史使用量排序
- AI 推荐：基于用户画像 + 场景 + 实时行为进行个性化排序

### 3.3 样本量估算方法（简要）

1. 明确核心指标：如推荐入口 CTR 或模板使用转化率
2. 确定期望提升幅度：例如希望 AI 推荐相对于热门推荐 CTR 提升 5%-10%
3. 选择显著性水平和检验功效：
   - 显著性水平 α = 0.05
   - 检验功效 1-β = 0.8
4. 利用二项分布近似或现有样本量计算工具（如 statsmodels）估算每组所需用户数 N
5. 实验期间根据实时数据复核样本量假设，避免过早停止

### 3.4 实验周期与显著性检验

- 实验周期建议：
  - 至少覆盖 2 个完整业务周期（例如 2 周），以平滑工作日/周末、节假日波动
- 检验方法：
  - 对 CTR、转化率等比例指标使用双样本比例检验（Z-test）
  - 如有连续型指标（如会话时长），可使用 t 检验或非参数检验

### 3.5 灰度发布计划

1. 阶段 1：小流量验证
   - 5%-10% 流量进入 AI 推荐组，监控系统稳定性和关键指标
2. 阶段 2：扩大实验
   - 将 AI 组提升至 20%-30%，同时保留对照组
3. 阶段 3：评估与决策
   - 若 AI 推荐在核心指标和安全指标（错误率、响应时间等）上均有显著优势，则逐步扩大覆盖
4. 阶段 4：全量上线与回滚预案
   - 保留 feature flag，出现异常时可快速回退到热门推荐策略

## 4. 数据采集需求

### 4.1 埋点数据

需要采集的关键事件及属性示例：

1. 推荐曝光事件：`recommendation_impression`
   - 属性：`user_id`, `session_id`, `experiment_group`, `candidate_templates`, `position`, `scene`（如首页 / 新建问卷弹窗）

2. 推荐点击事件：`recommendation_click`
   - 属性：`user_id`, `template_id`, `position`, `experiment_group`, `scene`

3. 模板创建与发布事件：`template_create`, `survey_publish`
   - 属性：`user_id`, `template_id`, `survey_id`, `scene`, `use_duration`（从进入编辑到发布的时间）

4. 问卷收集事件：`survey_response_reach`
   - 属性：`user_id`, `survey_id`, `response_cnt`, `reach_threshold_flag`

5. 用户反馈事件：`recommendation_feedback`
   - 属性：`user_id`, `template_id`, `feedback_type`（like/dislike）、可选文本反馈

### 4.2 数据存储与更新频率

- 行为日志存储：
  - 统一写入埋点日志系统（如 Kafka → 数据仓库/Hive/ClickHouse）
  - 保留较长时间窗口（>= 6 个月），支持回溯和模型回放

- 用户画像表：
  - 离线计算任务（如每日/每小时）生成 `user_profile` 宽表：
    - `user_id`, 行业、规模、角色
    - 使用频率特征（30/90 天）
    - 问卷创作和样本量特征
    - 偏好场景与题型特征
    - 分群结果 `user_cluster_id` 及标签
  - 对实时性要求高的特征（如最近一次活跃时间）可通过 Redis 等在线特征存储提供

- 实验与指标表：
  - `ab_experiment_assignment`：记录用户分组与实验元信息
  - `ab_metrics_daily`：按实验组、日期聚合 CTR、转化率等

---

## 5. 小结

本方案为问卷星推荐引擎提供：
- 一套可落地的用户画像特征体系（账号属性 + 使用行为 + 场景偏好）；
- 覆盖离线与在线阶段的评估指标体系；
- 包含分组策略、样本量估算思路与显著性检验的 A/B 测试方案；
- 清晰的数据采集与画像更新机制，可支撑后续推荐模型迭代与运营分析。

后续可结合真实数据模型与业务方反馈，对特征细节、标签命名和实验配置进行进一步细化与落地实施。
